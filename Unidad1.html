¿Qué es una base de datos?

Una base de datos es una colección de información organizada de forma que un programa de ordenador pueda seleccionar rápidamente los fragmentos de datos que necesite. Las bases de datos tradicionales se organizan por campos, registros y archivos.

Ficheros de datos.

Las bases de datos tradicionales se organizan por campos, registros y archivos.

  • Archivo de datos delimitados(csv).
  • Archivo de datos de ancho fijo.
  • Archivo de datos separados por tabulaciones.
  • Archivo XML.
  • Archivo JSON.

    • Archivo de datos delimitados.

    Los archivos CSV (comma-separated values) son un tipo de documento en formato de texto simple que representar datos en forma de tabla, en las que las columnas se separan por comas (o punto y coma en los países donde el separador decimal es la coma) y las filas o registros por saltos de línea.

    Archivo de ancho fijo.

    Los archivos de ancho fijo son archivos de texto en los que las columnas no están separadas por ningún delimitador de caracteres, como en el caso de los archivos CSV, sino más bien tener una longitud de caracteres fija (ancho). Los datos se suelen rellenar con espacios en blanco.

    Archivo de datos separados por tabulaciones.

    En este tipo de archivos, lo que separa cada uno de los valores o campos es un carácter de tabulación. El cambio de registro o entidad está determinado por un salto de línea como en los formatos anteriores.

    Archivo XML

    XML es el acrónimo de Extensible Markup Language, es decir, es un lenguaje de marcado que define un conjunto de reglas para la codificación de documentos.

    Archivo JSON

    JSON (acrónimo de JavaScript Object Notation, notación de objeto de JavaScript) es un formato de texto sencillo para el intercambio de datos.

    Las bases de datos.

    Concepto de base de datos.

    Las bases de datos son conjuntos o colecciones de datos que servirán tanto a usuarios y cómo a programas para definir, capturar, almacenar y consultar dichos datos. Para poder gestionar estas colecciones de datos de la manera de más óptima posible son necesarios los DBMS (Database Management System), en español SGBD (Sistema de Gestión de Bases de Datos).

    Objetivos de un buen DBMS

  • Evitar la redundancia de datos: El mismo cliente almacenado dos veces.
  • Evitar la inconsistencia: Si un cliente está almacenado dos veces, puede ser que existan datos distintos en cada registro, por ejemplo: distinto domicilio.
  • Evitar la dificultad para tener acceso a los datos: El acceso a los datos debe ser sencillo tanto para humanos cómo para programas. Además se debe gestionar el nivel de permisos de cada usuario.
  • Garantizar el aislamiento de datos: Es una de las cuatro propiedades ACID (Atomicidad, Consistencia, Aislamiento y Durabilidad) que deben cumplir las bases de datos durante una transacción.
  • Evitar las anomalías de acceso compartido: Varias aplicaciones o usuario pueden estar intentando acceder a los mismos datos. El DBMS debe garantizar una buena gestión de bloqueo de registro o campo para evitar conflictos.
  • Evitar problemas de seguridad: El DBMS o sistema gestor de base de datos debe asegurar que las bases de datos puedan ser compartidas, estén protegidas frente a fallos y no tengan problemas de accesos indebidos. En cuanto a la seguridad hay que considerar los siguientes aspectos:
  • Confidencialidad: Los datos deberían ser inaccesibles para las personas o programas no autorizadas.
  • Integridad: Los datos deben ser verídicos y fiables. Este aspecto se tratará más adelante.
  • Disponibilidad: No debe haber pérdida de información y los datos siempre tienen que estar disponibles.

    • Integridad de datos.

    Las violaciones de integridad vienen dadas, normalmente, por tipos de datos incorrectos o por duplicados, o eliminación de un registro del cual depende otro.

    Tipos de integridad que debe asegurar un DBMS

  • Integridad de dominio: los datos que se incluyen en cada campo dentro de un registro forman parte de un dominio (conjunto de valores posible). El dominio es el rango de valores posibles para un determinado campo.
  • Integridad de entidad: cada registro o entidad debe tener una clave primaria que lo identifique inequívocamente.
  • Integridad referencial: cuando un registro está asociado a otro o depende de otro, hay que asegurar que siempre existirá el registro del que depende. Por ejemplo: una factura siempre dependerá de la existencia de un cliente.

    • Modelos de bases de datos

    Se trata de una estructura lógica en la que la base de datos será construida, y que determinará cómo son almacenados.

    De momento trataremos solamente con el mas típico, que es el MER(modelo entidad-relación).

    Etapas de diseño de una base de datos

  • Diseño conceptual: Es una primera aproximación a la base de datos representada en un tipo de diagrama. El más utilizado es el modelo-entidad-relación.
  • Diseño lógico: Una vez definido el diseño conceptual se trasladará, teniendo en cuenta unas reglas determinadas, al esquema lógico, en el que se definirán tablas, columnas, asociaciones, etc.
  • Diseño físico: En este modelo se crearán en el DBMS los elementos que se han definido en el diseño lógico.

    • Conceptos de los modelos relacionales de bases de datos:

  • ENTIDAD: Se trata de un elemento del mundo real como puede ser un cliente.También se puede definir como registro fila.
  • ATRIBUTO O CAMPO: Propiedad o característica de una entidad.
  • RELACION: Conjunto de entidades del mismo tipo.
  • ASOCIACIÓN: Unión que existe entre 2 entidades.
  • CLAVE PRINCIPAL: También conocida como Primary Key(PK), es un atributo que identifica inequivocamente a una entidad. Debe ser única, no nula y, preferiblemente, que ocupe poco espacio.
  • CLAVE EXTRANJERA: También conocida como Foreign Key(FK), es cuando una entidad cede su PK a otra entidad para establecer una asociación.
  • TIPOS DE ASOCIACIONES:
  • Uno a uno(1:1): Una entida con otra entidad, por ejemplo, una persona con una partida de nacimiento.
  • Uno a muchos(1:N): Una entidad con varias entidades, por ejemplo, un cliente a varias facturas.
  • Muchos a 1(N:1): Varias entidades con 1 entidad, al revés que la anterior.
  • Muchos a muchos(N:M): Varias entidades con varias entidades, por ejemplo, un producto a muchas facturas, y una factura con muchos productos. Una vez que queremos hacer el diseño físico, son inviables con lo cual tenemos que transformarlas en 2 relaciones 1:N creando una entidad asociativa que las una.

    Base de datos para una ferretería en la que queremos saber las relaciones entre los clientes, las facturas y los productos. Por un lado tenemos el modelo entidad relación, y por otro el diseño lógico.
